## 1. Краткое описание проекта

Проект предусматривает разработку мобильного приложения для автоматизации прямой доставки товаров в точки продаж (Direct Store Delivery). Приложение предназначено для торговых представителей, экспедиторов и водителей-экспедиторов, осуществляющих доставку товаров со склада непосредственно в розничные точки с одновременным оформлением всех сопроводительных документов.

Ключевые характеристики решения: полноценная работа в offline-режиме с последующей синхронизацией данных, глубокая интеграция с 1С:ERP в качестве backend-системы, поддержка платформ Android и iOS, а также работа с периферийным оборудованием (мобильные принтеры, сканеры штрихкодов).

### 1.1. Цели проекта

- Сократить время оформления доставки на 40–60% за счёт мобильной автоматизации
- Обеспечить 100% работоспособность приложения вне зоны покрытия сети
- Исключить двойной ввод данных путём прямой интеграции с 1С:ERP
- Обеспечить контроль и отслеживание статусов доставок в реальном времени (при наличии связи)
- Снизить количество ошибок при приёмке и отгрузке товаров на 80%

## 2. Роли пользователей

Приложение поддерживает три основные роли с различным уровнем доступа и набором функций. Каждая роль определяет видимость экранов, доступные операции и объём данных, синхронизируемых на устройство.

### 2.1. Экспедитор (Водитель-экспедитор)

**Описание:** Основной пользователь приложения. Осуществляет физическую доставку товаров со склада в розничные торговые точки, оформляет все сопроводительные документы, принимает оплату и обрабатывает возвраты.

**Бизнес-процесс рабочего дня:**

1. Авторизация в приложении, получение маршрутного листа на день
2. Приёмка товара на борт ТС на складе (загрузка рейса) со сканированием
3. Движение по маршруту с последовательным посещением торговых точек
4. На каждой точке: отгрузка товара, приём возвратов/тары, приём оплаты, фотофиксация
5. Завершение рейса: ревизия остатков, сдача наличных (инкассация), закрытие дня

**Доступные функции:**

| Функциональный блок | Операции |
|---------------------|----------|
| Маршрутный лист | Просмотр списка точек, навигация, отметка визита (прибытие/убытие), добавление внепланового визита |
| Загрузка рейса | Сканирование товаров при загрузке в ТС, подтверждение принятого количества, фиксация расхождений с заданием на загрузку |
| Отгрузка товара | Сканирование при выгрузке, оформление накладной, частичная доставка, фиксация расхождений, электронная подпись получателя |
| Возвраты | Оформление возврата товара (по качеству, по сроку годности, невостребованный), сканирование, фотофиксация дефектов |
| Возвратная тара | Приём пустой тары от точки, сканирование/ручной ввод количества, оформление акта приёма тары |
| Приём оплаты | Приём наличных с расчётом сдачи, регистрация безналичной оплаты, формирование ПКО |
| Складские операции | Ревизия остатков в кузове, перемещение товара другому экспедитору |
| Инкассация | Сдача собранных наличных, формирование акта инкассации |
| Печать | Печать накладных, актов, ПКО на мобильном Bluetooth-принтере |
| Фотоотчёты | Фотографирование полки, выкладки, витрины (опционально) |

**Ограничения роли:**

- Не видит данных других экспедиторов и их маршрутов
- Не может изменять справочники (номенклатура, контрагенты, цены)
- Не может создавать заказы покупателей (только отгрузка по существующим)
- Не имеет доступа к аналитическим дашбордам и отчётам по другим сотрудникам
- Не может менять настройки приложения и параметры синхронизации

**Синхронизируемые данные:** маршрутный лист на день, справочник номенклатуры (только позиции маршрута), контрагенты маршрута, цены, остатки загруженного товара, задолженность по точкам маршрута.

---

### 2.2. Супервайзер

**Описание:** Руководитель группы экспедиторов и/или торговых представителей. Контролирует выполнение планов доставки, мониторит местоположение полевых сотрудников, анализирует результативность и принимает оперативные решения по перераспределению задач.

**Бизнес-процесс рабочего дня:**

1. Авторизация, просмотр сводной панели за текущий день
2. Контроль выхода экспедиторов на маршрут (факт загрузки рейса)
3. Мониторинг прогресса выполнения маршрутов в реальном времени
4. Обработка исключительных ситуаций (отказы, недопоставки, задержки)
5. Анализ результатов дня, формирование отчётов

**Доступные функции:**

| Функциональный блок | Операции |
|---------------------|----------|
| Дашборд реального времени | Сводка по всем экспедиторам: статус (на маршруте / на точке / завершил), процент выполнения плана, собранные оплаты |
| Карта мониторинга | GPS-позиции всех подчинённых экспедиторов на карте в реальном времени, треки перемещений, отклонения от маршрута |
| Маршрутные листы | Просмотр маршрутов всех подчинённых, статус каждого визита, время на точке, фактическое время vs плановое |
| Управление маршрутами | Переназначение точки другому экспедитору, добавление/удаление точки из маршрута, изменение приоритета |
| Документы доставки | Просмотр всех оформленных накладных, актов, возвратов в режиме только чтение |
| Аналитика и отчёты | Отчёт по выполнению плана доставок (день/неделя/месяц), отчёт по возвратам, отчёт по собранным оплатам, отчёт по времени на маршруте |
| Возвраты и рекламации | Просмотр всех возвратов, утверждение/отклонение возвратов сверх лимита, анализ причин возвратов по точкам и номенклатуре |
| Уведомления | Алерты: экспедитор не начал маршрут, длительная задержка на точке, отклонение от маршрута, крупный возврат, превышение лимита задолженности |
| Сообщения | Отправка текстовых сообщений экспедиторам через приложение |

**Ограничения роли:**

- Не может оформлять документы доставки (только просмотр)
- Не может принимать оплату и проводить инкассацию
- Не может изменять справочники, цены и ценовые политики
- Не может управлять учётными записями пользователей
- Видит данные только своей группы экспедиторов (определяется в 1С)

**Синхронизируемые данные:** маршрутные листы всех подчинённых, GPS-позиции экспедиторов, документы доставки (только заголовки + итоги), агрегированная аналитика, справочник контрагентов всех маршрутов группы.

### 2.3. Администратор

**Описание:** Системный администратор мобильного решения. Управляет пользователями, настройками приложения, параметрами синхронизации и контролирует работоспособность системы. Как правило, это сотрудник IT-отдела или оператор 1С с расширенными правами.

**Бизнес-процесс:**

Администратор не привязан к ежедневному маршрутному циклу. Его задачи — настройка и поддержка системы, управление доступом, мониторинг технического состояния, разрешение нештатных ситуаций.

**Доступные функции:**

| Функциональный блок | Операции |
|---------------------|----------|
| Управление пользователями | Создание/блокировка/удаление учётных записей, назначение ролей (экспедитор/супервайзер), привязка к группе/складу, сброс пароля, управление сессиями |
| Управление устройствами | Реестр зарегистрированных устройств, привязка устройства к пользователю, удалённая блокировка и очистка данных, просмотр версии приложения на устройстве |
| Настройки синхронизации | Частота синхронизации справочников, максимальный срок работы в offline, лимит объёма данных на устройстве, приоритеты синхронизации (справочники vs документы vs медиа) |
| Мониторинг системы | Статус синхронизации по каждому устройству, журнал ошибок обмена, очередь неотправленных документов, метрики производительности Middleware |
| Справочники (чтение) | Просмотр номенклатуры, контрагентов, ценовых политик (редактирование — только в 1С) |
| Разрешение конфликтов | Просмотр конфликтов синхронизации, ручное разрешение (принять серверную/мобильную версию), повторная отправка отклонённых документов |
| Журнал аудита | Полный лог действий всех пользователей, история входов/выходов, попытки несанкционированного доступа |
| Настройки печати | Управление шаблонами печатных форм, привязка принтеров к пользователям |
| Обновление приложения | Push-уведомление о доступности новой версии, контроль минимально допустимой версии, принудительное обновление |

**Ограничения роли:**

- Не может оформлять бизнес-документы (доставки, возвраты, оплаты)
- Не может изменять мастер-данные в 1С (только просмотр через приложение)
- Не может переназначать маршруты (это функция супервайзера)

**Синхронизируемые данные:** метаданные всех пользователей, статусы устройств, журналы синхронизации, агрегированные метрики системы. Администратор не получает на устройство бизнес-данные (товары, цены, документы доставки).

## 3. Функциональные требования

### 3.1. Управление маршрутами и визитами

- Получение маршрутных листов из 1С:ERP с последовательностью визитов
- Отображение маршрута на карте с навигацией (поддержка Яндекс.Карт и OSM)
- Оптимизация маршрута с учётом временных окон и приоритетов точек
- Регистрация факта визита: время прибытия/убытия, GPS-координаты, фотофиксация
- Возможность добавления внеплановых визитов и корректировки маршрута

### 3.2. Оформление документов доставки

- Создание актов приёмки-передачи товаров с поддержкой сканирования штрихкодов
- Оформление накладных, счетов-фактур, УПД
- Обработка частичной доставки: приёмка с расхождениями, недопоставка, пересортица
- Электронная подпись получателя на экране устройства
- Печать документов на мобильном Bluetooth-принтере

### 3.3. Приём заказов и торговые операции

- Каталог товаров с фотографиями, ценами, остатками и описаниями
- Оформление заказов с учётом ценовых политик и скидок из 1С
- Контроль лимитов дебиторской задолженности в режиме реального времени
- Просмотр истории заказов и отгрузок по точке
- Работа с промо-акциями и специальными условиями

### 3.4. Финансовые операции

- Приём оплаты наличными с расчётом сдачи и формированием ПКО
- Регистрация безналичных платежей (QR-код, эквайринг)
- Инкассация: учёт собранных денежных средств, акт инкассации
- Контроль дебиторской задолженности по каждой торговой точке

### 3.5. Складские операции на маршруте

- Приёмка товара на борт транспортного средства (загрузка рейса)
- Учёт остатков в кузове / на борту в течение рейса
- Ревизия остатков по завершении рейса с формированием акта
- Перемещение товаров между экспедиторами

### 3.6. Оформление возвратов товара

#### 3.6.1. Виды возвратов

Приложение поддерживает следующие виды возвратов, каждый из которых имеет свой бизнес-процесс и набор обязательных полей:

| Вид возврата | Описание | Обязательные данные | Документ в 1С |
|-------------|----------|---------------------|---------------|
| Возврат по качеству (брак) | Товар повреждён, дефект упаковки, нарушение целостности | Фото дефекта (мин. 1 шт.), причина из справочника, номер партии/серия | Возврат товаров от клиента (причина: брак) |
| Возврат по сроку годности | Товар с истёкшим или критически коротким сроком годности | Фото маркировки с датой, дата производства/срока годности, номер партии | Возврат товаров от клиента (причина: срок годности) |
| Невостребованный товар | Точка отказывается от части или всей поставки | Причина отказа из справочника, подпись представителя точки | Возврат товаров от клиента (причина: отказ) |
| Пересортица | Поставлен не тот товар, который заказывали | Фактический и ожидаемый товар, ссылка на исходный заказ | Возврат + корректировка реализации |
| Возврат по рекламации | Возврат ранее принятого товара по рекламации от точки | Номер исходной накладной, описание проблемы, фото | Возврат товаров от клиента (причина: рекламация) |

#### 3.6.2. Бизнес-процесс оформления возврата

1. **Инициация возврата.** Экспедитор выбирает точку и открывает экран «Возврат товара». Система отображает историю поставок в эту точку (последние 30 дней, настраивается).

2. **Выбор товара для возврата.** Экспедитор может: выбрать товар из списка предыдущих поставок (с привязкой к накладной), отсканировать штрихкод товара, добавить товар вручную из каталога.

3. **Указание причины и количества.** Для каждой позиции возврата указывается: причина возврата (из справочника причин, синхронизируемого из 1С), количество единиц возврата, состояние товара (годен для перепродажи / не годен / требует проверки).

4. **Фотофиксация.** Для возвратов по качеству и сроку годности — фотографирование обязательно. Фотографии привязываются к конкретной строке возврата. Минимальное количество фото настраивается в 1С (по умолчанию: 1 для брака, 1 для срока годности).

5. **Подтверждение и подпись.** Экспедитор просматривает итоговый акт возврата, получает электронную подпись представителя торговой точки на экране устройства.

6. **Печать и сохранение.** Печать акта возврата на мобильном принтере (2 экземпляра). Документ сохраняется в локальной БД и встаёт в очередь синхронизации.

7. **Синхронизация и обработка в 1С.** При появлении связи документ отправляется в 1С. 1С создаёт документ «Возврат товаров от клиента», при необходимости — корректировку реализации. Результат обработки возвращается на устройство (подтверждение / отклонение с причиной).

#### 3.6.3. Правила и ограничения возвратов

- **Лимит возврата без утверждения супервайзером.** Настраиваемый порог (по сумме и/или количеству). При превышении лимита документ создаётся со статусом «На утверждении» и отправляется супервайзеру.
- **Привязка к исходной поставке.** Для возвратов по рекламации обязательна привязка к номеру исходной накладной. Приложение контролирует, что возвращаемое количество не превышает поставленное.
- **Срок возврата.** Настраиваемый максимальный срок возврата (по умолчанию — 30 дней с даты поставки). Попытка оформить возврат по истечении срока требует утверждения супервайзером.
- **Статусная модель:** Создан → На утверждении (если требуется) → Утверждён → Синхронизирован → Обработан в 1С / Отклонён.

### 3.7. Оформление возвратной тары

#### 3.7.1. Виды возвратной тары

| Вид тары | Примеры | Единица учёта | Особенности |
|---------|---------|---------------|-------------|
| Пластиковые ящики/лотки | Хлебные лотки, ящики для бутылок | Штука | Учёт по типу тары, без серийных номеров |
| Стеклянная тара (бутылки) | Бутылки 0.33л, 0.5л, 1л | Штука (по типоразмеру) | Учёт по типоразмеру, может быть залоговая стоимость |
| Кеги | Кеги 30л, 50л | Штука (с серийным номером) | Обязательный серийный учёт, высокая стоимость |

#### 3.7.2. Бизнес-процесс приёма возвратной тары

1. **Начало приёма тары.** На экране визита в торговой точке экспедитор открывает раздел «Приём тары». Система отображает ожидаемое количество тары по данной точке (на основании предыдущих поставок и баланса тары из 1С).

2. **Фиксация фактического количества.** Экспедитор вводит фактическое количество тары по каждому виду: сканирование штрихкода/QR-кода тары (если маркирована), ручной ввод количества по виду тары, при серийном учёте (кеги, паллеты) — сканирование серийного номера каждой единицы.

3. **Оценка состояния.** Для каждого вида тары указывается состояние: «Годная» (может быть повторно использована), «Повреждённая» (с указанием характера повреждения), «Списание» (не подлежит использованию).

4. **Расхождения.** Если фактическое количество отличается от ожидаемого, система фиксирует расхождение. При расхождении свыше настраиваемого порога (по умолчанию — 10%) требуется фотофиксация и комментарий.

5. **Формирование акта.** Приложение автоматически формирует «Акт приёма возвратной тары», включающий: дату и время, точку и контрагента, перечень тары с количеством по видам и состоянию, расхождения с ожидаемым количеством, подпись экспедитора и представителя точки.

6. **Загрузка тары в ТС.** Принятая тара учитывается в остатках борта ТС. При ревизии по завершении рейса тара учитывается отдельно от товара.

7. **Синхронизация с 1С.** Акт приёма тары синхронизируется как отдельный документ. В 1С создаётся документ «Перемещение тары» (или аналогичный регистр учёта тары), обновляется баланс тары по контрагенту.

#### 3.7.3. Баланс тары по контрагенту

Приложение отображает экспедитору текущий баланс тары по каждой торговой точке:

- **Передано тары** — суммарное количество тары, отгруженной в точку (с товаром)
- **Возвращено тары** — суммарное количество тары, принятой обратно
- **Сальдо** — разница (задолженность точки по таре)
- **Залоговая стоимость** — денежный эквивалент задолженности по таре (если используется залоговая схема)

Данные по балансу тары синхронизируются из 1С при получении маршрутного листа. Экспедитор видит баланс при входе на экран визита в точку и может использовать его при переговорах о возврате тары.

#### 3.7.4. Интеграция тары с 1С:ERP

| Операция | Триггер | Документ/объект в 1С |
|----------|---------|----------------------|
| Отгрузка тары с товаром | Оформление доставки | Табличная часть «Тара» в документе «Реализация товаров и услуг» |
| Приём тары от точки | Оформление приёма тары | Документ «Возврат тары от покупателя» или регистр сведений «Движения тары» |
| Списание повреждённой тары | Фиксация повреждения | Документ «Списание тары» |
| Баланс тары по контрагенту | Синхронизация справочников | Регистр накопления «Тара у контрагентов» |
| Залоговая стоимость | Расчёт задолженности | Регистр взаиморасчётов по таре |

### 3.8. Мерчандайзинг (опционально)

- Фотоотчёты с привязкой к планограммам
- Аудит полки: наличие товаров, ценников, POS-материалов
- Задания мерчандайзинга с чек-листами

#### 4.1.1. Экран авторизации

**Роли:** все.

При отсутствии сети — вход по ранее сохранённым credentials (offline-авторизация с PIN-кодом). Отображает индикатор состояния связи и версию приложения.

#### 4.1.2. Экран быстрого входа (PIN)

**Роли:** все.

После первичной авторизации повторный вход осуществляется через 4-значный PIN или биометрию. Три неудачные попытки блокируют PIN и требуют полную авторизацию.

**Роли:** все (содержимое адаптируется по роли).

**Супервайзер:** сводная панель — количество экспедиторов на маршруте, общий процент выполнения, проблемные точки (возвраты, задержки). Быстрые кнопки: «Карта мониторинга», «Отчёты», «Сообщения».

**Администратор:** статус системы — количество активных устройств, ошибки синхронизации за последние 24 ч, очередь необработанных документов. Быстрые кнопки: «Пользователи», «Устройства», «Журнал».

#### 4.1.4. Экран настроек

**Роли:** все (состав настроек по роли).

Профиль пользователя, смена PIN-кода, выбор принтера (экспедитор), настройка уведомлений, информация о версии приложения, ручной запуск синхронизации, переключение языка (при мультиязычности), экран «О приложении».

#### 4.1.5. Экран уведомлений

**Роли:** все.

Лента уведомлений с фильтрацией по типу: системные (синхронизация, обновления), бизнесовые (новый заказ, изменение маршрута), сообщения от супервайзера. Непрочитанные выделяются визуально, поддержка pull-to-refresh.

**Роли:** Экспедитор.

Построение оптимального маршрута, кнопка «Навигация» (открывает внешнее навигационное приложение или встроенную навигацию). Поддержка оффлайн-карт (предзагруженные тайлы для региона маршрута).

Центральный экран рабочего процесса. Открывается при нажатии на точку в маршруте. Содержит: информацию о точке (название, адрес, контакт, график работы), кнопку «Начать визит» (фиксирует время прибытия и GPS), вкладки/секции действий: «Отгрузка», «Возвраты», «Тара», «Оплата», «Фотоотчёт». Индикатор задолженности точки (сумма и просрочка). Баланс тары по точке. Кнопку «Завершить визит» (фиксирует время убытия, проверяет заполненность обязательных данных).

#### 4.2.4. Экран отгрузки товара

**Роли:** Экспедитор.

Список товаров к отгрузке по данной точке (из заказа). Для каждой позиции: наименование, артикул, план (заказ), факт (поле ввода), единица измерения, цена, сумма. Кнопка сканирования штрихкода для быстрого поиска товара и подтверждения отгрузки. Отметка расхождений (недовоз, пересортица). Итого по документу: количество, сумма, НДС. Кнопка «Подпись получателя» — переход на экран электронной подписи. Кнопка «Печать накладной».

**Роли:** Экспедитор (создание), Супервайзер (просмотр/утверждение).

Форма оформления возврата. Выбор товара: из списка предыдущих поставок, сканированием штрихкода или из каталога. Для каждой позиции: наименование, количество возврата, причина (выбор из справочника), состояние товара (годен / не годен / на проверку), кнопка «Сделать фото». Секция фотографий — превью прикреплённых фото с возможностью удаления. Блок комментария (свободный текст). Итого: количество позиций, общая сумма возврата. Индикатор «Требуется утверждение супервайзером» (если превышен лимит). Кнопка «Подпись» и «Печать акта возврата».

#### 4.2.6. Экран приёма возвратной тары

**Роли:** Экспедитор (создание), Супервайзер (чтение).

Отображает баланс тары по текущей точке (сверху экрана). Список видов тары с колонками: вид тары, ожидаемое количество (из 1С), фактическое количество (поле ввода), состояние (годная / повреждённая / списание), расхождение (рассчитывается автоматически). Кнопка сканирования для маркированной тары. При серийном учёте (кеги, паллеты) — экран ввода серийных номеров (поочерёдное сканирование). Блок «Повреждённая тара» — фото и описание повреждения. Итого: принято годной, принято повреждённой, к списанию. Кнопка «Сформировать акт», «Подпись», «Печать».

#### 4.2.7. Экран приёма оплаты

**Роли:** Экспедитор.

Информация о задолженности точки (текущая + просроченная). Выбор типа оплаты: наличные / безналичные / QR. Для наличных: поле ввода полученной суммы, автоматический расчёт сдачи. Для безналичных: поле ввода суммы, выбор способа (перевод / эквайринг / QR). Формирование ПКО (приходного кассового ордера). Печать квитанции. Кнопка «Подтвердить оплату».

#### 4.2.8. Экран загрузки рейса

**Роли:** Экспедитор.

Фиксация расхождений (план vs факт). Дополнительно: секция загрузки тары (пустая тара для передачи в точки). Кнопка «Подтвердить загрузку» — формирует акт загрузки рейса. Фиксация номера ТС и показаний одометра (опционально).

#### 4.2.9. Экран ревизии остатков

**Роли:** Экспедитор.

Отображает расчётные остатки товара в кузове (на основании загрузки минус отгрузки плюс возвраты). Поле ввода фактического количества по каждой позиции. Сканирование штрихкодов для подтверждения. Отдельная секция: ревизия тары (расчётный остаток тары в кузове vs факт). Фиксация расхождений с указанием причины. Формирование акта ревизии. Используется в конце рейса перед инкассацией.

#### 4.2.10. Экран инкассации

**Роли:** Экспедитор.

Отображает общую сумму собранных за рейс наличных (по точкам). Детализация по точкам: наименование, сумма, номер ПКО. Поле ввода фактической суммы при сдаче. Фиксация расхождений. Формирование акта инкассации. Печать акта.

#### 4.2.11. Экран электронной подписи

**Роли:** Экспедитор (инициирует).

Полноэкранное поле для рукописной подписи пальцем/стилусом. Отображение ФИО подписанта (вводится вручную или выбирается из контактных лиц точки). Кнопки: «Очистить», «Подтвердить». Подпись сохраняется как изображение и привязывается к документу.

#### 4.2.12. Экран сканирования

**Роли:** Экспедитор.

Экран камеры для сканирования штрихкодов (EAN-13, EAN-8, Code128, QR). Подсветка (фонарик). Звуковой сигнал при успешном сканировании. Режим непрерывного сканирования (для загрузки рейса) и режим одиночного сканирования. Отображение результата сканирования: наименование товара, текущее количество.

#### 4.4.4. Разрешение конфликтов

**Роли:** Администратор.

Список конфликтов, требующих ручного разрешения. Для каждого: тип конфликта, устройство, пользователь, время, сравнение серверной и мобильной версии данных. Действия: принять серверную версию, принять мобильную версию, объединить (для некоторых типов). Ссылка на соответствующий документ в 1С.

#### 4.4.5. Журнал аудита

**Роли:** Администратор.

Полный лог действий пользователей: входы/выходы, создание/изменение документов, синхронизации, попытки доступа к чужим данным. Фильтры: по пользователю, по действию, по периоду, по статусу (успех / ошибка). Экспорт журнала в CSV.

#### 4.4.6. Настройки системы

**Роли:** Администратор.

Параметры синхронизации (периодичность, лимиты). Лимиты возвратов (сумма без утверждения, максимальный срок). Параметры offline-режима (макс. срок, лимит данных). Минимальная версия приложения (с принудительным обновлением). Шаблоны печатных форм. Настройки push-уведомлений.

## 5. Архитектура решения

### 5.1. Общая архитектура

Система строится по принципу offline-first с трёхзвенной архитектурой:

| Компонент | Технология | Назначение |
|-----------|-----------|------------|
| Мобильное приложение | React Native / Flutter | Клиентское приложение для полевых сотрудников |
| Middleware / API Gateway | Node.js (NestJS) или Go | Управление синхронизацией, очередями, маршрутизация запросов |
| Локальная БД на устройстве | SQLite / Realm / WatermelonDB | Хранение данных offline, очередь синхронизации |
| Backend интеграция | 1С:ERP + HTTP-сервисы | Мастер-система: справочники, документы, бизнес-логика |
| Очереди сообщений | RabbitMQ / Apache Kafka | Надёжная асинхронная доставка сообщений между Middleware и 1С |
| Сервер push-уведомлений | Firebase Cloud Messaging | Уведомления о новых заказах, изменениях маршрута |

**Схема взаимодействия:** Мобильное приложение ↔ Middleware (REST/GraphQL API) ↔ Очередь сообщений ↔ 1С:ERP HTTP-сервисы. Такой подход позволяет изолировать 1С от прямых запросов мобильных устройств, обеспечить буферизацию и retry-логику.

### 5.2. Offline-архитектура

Offline-режим является центральным архитектурным требованием. Приложение должно сохранять полную функциональность без подключения к сети.

#### 5.2.1. Стратегия локального хранения

- **Справочники** (товары, контрагенты, цены, маршруты) — полная копия релевантного подмножества данных на устройстве
- **Документы** — создаются локально с присвоением временных UUID, заменяемых серверными ID после синхронизации
- **Фотографии и медиа** — сохраняются локально, загружаются при появлении связи с приоритизацией
- **GPS-трек** — непрерывная запись в локальный буфер, пакетная выгрузка

#### 5.2.2. Механизм синхронизации

Синхронизация реализуется по принципу event sourcing с использованием следующих механизмов:

1. Каждое изменение данных записывается как событие (event) в локальный журнал операций
2. При появлении связи события отправляются на Middleware в хронологическом порядке
3. Middleware валидирует события, разрешает конфликты и транслирует их в 1С
4. 1С обрабатывает события и возвращает подтверждения с серверными ID
5. Подтверждения применяются к локальной БД, журнал очищается

#### 5.2.3. Разрешение конфликтов

| Тип конфликта | Стратегия | Пример |
|---------------|-----------|--------|
| Изменение справочника во время offline | Server wins (сервер главный) | Цена товара изменилась — применяется серверная цена |
| Одновременное редактирование заказа | Last write wins + уведомление | Два устройства изменили один заказ |
| Превышение лимита задолженности | Soft validation offline + hard validation online | Заказ создан offline, при синхронизации превышен лимит |
| Остаток товара недостаточен | Optimistic locking с компенсацией | Товар продан двумя экспедиторами — один заказ отклоняется |

## 6. Интеграция с 1С:ERP

### 6.1. Архитектура интеграции

Интеграция строится через HTTP-сервисы 1С:ERP, опубликованные через веб-сервер (Apache/IIS). Прямое подключение мобильных устройств к 1С не допускается — все взаимодействия проходят через Middleware.

### 6.2. Обмен данными

#### 6.2.1. Данные из 1С в мобильное приложение (Download)

| Объект 1С | Периодичность | Формат | Объём (оценка) |
|-----------|---------------|--------|----------------|
| Справочник Номенклатура | 1 раз в день + изменения | JSON (delta-sync) | 5 000–50 000 позиций |
| Справочник Контрагенты | 1 раз в день + изменения | JSON (delta-sync) | 500–5 000 записей |
| Прайс-листы и скидки | 1 раз в день | JSON | Матрица цен |
| Маршрутные листы | Начало рабочего дня | JSON | 10–40 точек на маршрут |
| Остатки на складе | Каждые 15–30 минут | JSON (delta) | По номенклатуре маршрута |
| Дебиторская задолженность | Каждые 30 минут | JSON | По контрагентам маршрута |
| Заказы покупателей | По событию | JSON + Push | Новые/изменённые заказы |
| Баланс тары по контрагентам | 1 раз в день + изменения | JSON | По контрагентам маршрута |
| Справочник причин возвратов | 1 раз в день | JSON | 10–30 записей |
| Справочник видов тары | 1 раз в день | JSON | 10–50 записей |

#### 6.2.2. Данные из мобильного приложения в 1С (Upload)

| Документ | Триггер отправки | Объект 1С |
|----------|------------------|-----------|
| Акт доставки | Завершение визита + синхронизация | Реализация товаров и услуг |
| Заказ покупателя | Оформление заказа + синхронизация | Заказ клиента |
| Возврат товаров | Оформление возврата + синхронизация | Возврат товаров от клиента |
| Акт приёма тары | Оформление приёма тары + синхронизация | Возврат тары от покупателя / Регистр движений тары |
| Списание тары | Фиксация повреждения тары | Списание тары |
| Приём оплаты | Приём наличных/безнал + синхронизация | Приходный кассовый ордер / Оплата |
| Акт инкассации | Завершение рейса | Приходный кассовый ордер (касса) |
| GPS-трек | Фоновая выгрузка | Регистр сведений (координаты) |
| Фотоотчёт | Фоновая выгрузка | Присоединённые файлы |

### 6.3. HTTP-сервисы 1С

На стороне 1С:ERP необходимо разработать следующий набор HTTP-сервисов:

- `/api/v1/auth` — аутентификация пользователя, получение JWT-токена
- `/api/v1/sync/references` — инкрементальная выгрузка справочников (delta sync по дате изменения)
- `/api/v1/sync/routes` — получение маршрутных листов на дату
- `/api/v1/sync/prices` — актуальные прайс-листы и условия скидок
- `/api/v1/sync/stock` — остатки товаров на складах
- `/api/v1/sync/debts` — дебиторская задолженность контрагентов
- `/api/v1/sync/packaging` — баланс тары по контрагентам, справочник видов тары
- `/api/v1/sync/return-reasons` — справочник причин возвратов
- `/api/v1/documents/upload` — приём пакета документов от мобильного приложения
- `/api/v1/documents/returns` — приём документов возвратов товаров
- `/api/v1/documents/packaging` — приём документов по движению тары
- `/api/v1/documents/status` — статус обработки загруженных документов
- `/api/v1/media/upload` — загрузка фотографий и медиафайлов

### 6.4. Обеспечение надёжности обмена

- **Idempotency keys** для всех операций записи — повторная отправка того же документа не создаёт дубликаты
- **Sequence numbering** — нумерация пакетов для обнаружения пропусков
- **Retry с exponential backoff** при ошибках сети
- **Dead letter queue** для необработанных сообщений с оповещением оператора
- **Полный аудит-лог** всех обменов (запрос, ответ, время, статус)

## 8. Безопасность

- **Аутентификация:** JWT-токены с refresh-механизмом (access token — 15 мин, refresh token — 30 дней)
- **Авторизация:** RBAC (Role-Based Access Control) с синхронизацией ролей из 1С
- **Шифрование данных на устройстве:** AES-256 для локальной базы (SQLCipher)
- **TLS 1.3** для всех сетевых взаимодействий с certificate pinning
- **Биометрическая аутентификация** (отпечаток пальца, Face ID) для разблокировки приложения
- **Удалённая блокировка и очистка данных** при утере устройства через MDM
- **Ограничение максимального срока работы в offline-режиме** (настраиваемое, по умолчанию 72 часа)
- **Логирование** всех действий пользователя для аудита

| Время оформления возврата | < 3 минут на документ | Аналитика приложения |

## 14. Заключение

Разработка мобильного DSD-приложения с offline-режимом и интеграцией с 1С:ERP — комплексный проект, успех которого зависит от правильного выбора архитектуры синхронизации, тщательного проектирования API-интеграции и глубокого понимания бизнес-процессов прямой доставки.

**Ключевые рекомендации:**

1. **Начинать с MVP:** маршруты, доставка, базовая синхронизация. Не пытаться реализовать всё сразу.
2. **Инвестировать в архитектуру offline-синхронизации** — это технически самая сложная и критически важная часть проекта.
3. **Использовать Middleware как буфер** между мобильным приложением и 1С — это защитит 1С от перегрузок и обеспечит гибкость.
4. **Проводить пилотное внедрение** на ограниченной группе маршрутов перед полным rollout.
5. **Обеспечить тесное взаимодействие** между мобильной командой и разработчиками 1С на всех этапах проекта.
6. **Уделить особое внимание учёту тары** — расхождения в учёте тары являются типичной проблемой DSD-систем и требуют регулярных сверок и ревизий.

При правильной реализации проект позволит существенно повысить эффективность прямой доставки, снизить операционные расходы и обеспечить прозрачность процессов для всех участников цепочки поставок.
