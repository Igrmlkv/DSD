## 1. Дорожная карта по версиям

```
   v1: опрос мерчендайзера + фото-evidence
   ──────────────────────────────────────────────────►
                  v2: Trax SDK / альтернативы (открытый вендор)
                  ─────────────────────────────────────►
                                    v3: собственный CV-стек on-prem
                                    ─────────────────────────►
   2026 Q2 ─ Q3                2026 Q4 ─ 2027 Q2          2027 Q3 ─ 2028
```

| Версия | Цель | Длительность | Готовность инфраструктуры |
| --- | --- | --- | --- |
| **v1** | Заменить бумажные чек-листы цифровым опросом + фото; первая витрина PSS | 3–4 мес | Только DSD-middleware (без ML) |
| **v2** | Сократить ручной ввод до 50% за счёт автозаполнения от вендорского CV-SDK | 4–6 мес | + интеграция с CV-вендором (cloud или on-prem) |
| **v3** | Полная независимость от внешнего вендора; собственная модель под локальный ассортимент | 9–12 мес | + on-prem k8s + GPU + MLOps |

## 2. Фаза 0 — Подготовка (4–6 недель)

**Цель.** Финализировать ТЗ, согласовать архитектуру, закрыть все 7 открытых вопросов спецификации (§13 ТЗ), выбрать пилотный регион.

### 2.1. Закрытие открытых вопросов спецификации (§13 ТЗ)

| № | Вопрос ТЗ | Целевое решение в фазе 0 | Ответственный |
| --- | --- | --- | --- |
| Q1 | Окончательный список SKU для MVP | Согласованный список 50–100 топ-SKU (приоритет, новые, фокус-бренды) | Trade-marketing + бренд-команды |
| Q2 | Объём и качество эталонных фото | Аудит фото-архива; план съёмки эталонов (≥3 ракурса × ≥100 SKU MVP) | ИТ + студия |
| Q3 | Финальный вендор v2 | Запуск RFP; не блокирует фазу 1 | Закупки + ИТ-архитектор |
| Q4 | On-prem vs гибрид для обучения v3 | Зафиксированная позиция: inference on-prem обязательно; обучение допустимо в изолированном VPC | InfoSec + ИТ |
| Q5 | SLA ручной модерации v2/v3 | Целевой норматив (рекомендация: ≥80% фото за 24 ч) | Бизнес + операционный директор |
| Q6 | Температура холодильника по фото | GO/NO-GO по OCR термометра (зависит от наличия дисплея в кадре по типовой ТТ) | Бизнес + ML-команда |
| Q7 | ROI-модель (cost per audit vs экономия времени) | Утверждённая финансовая модель с целевыми KPI окупаемости | CFO + PM |

### 2.2. Дополнительные активности

- Финальное согласование `docs/merchandising_module_spec.md` со стейкхолдерами (бизнес, ИТ, безопасность, юристы, InfoSec).
- Утверждение пилотного региона (рекомендуется регион с большим разнообразием каналов: ритейл + киоски + 1–2 HoReCa-сети) и состава команды (5–10 мерчендайзеров, 1 супервайзер).
- Настройка инфраструктуры разработки: репозитории, ci/cd, dev/staging/prod environments middleware.
- **Аудит парка устройств** (см. §9 ТЗ): подтверждение Android 10+ / iOS 14+, камера ≥ 8 Мп, 3 ГБ ОЗУ; план замены устройств ниже минимума.
- Аудит существующего фото-архива (для обучения v3 в будущем); если эталонов нет — запуск студийной съёмки параллельно с фазой 1.
- Подписание DPA / NDA с потенциальными CV-вендорами; запуск **RFP** для v2 (см. §4.1).
- Замер базового времени бумажного аудита по типовым ТТ (для KPI пилота — «время аудита ≤ 70% от бумажного»).
- Привлечение InfoSec для ревью архитектуры и согласования DPIA по 152-ФЗ (§8.7, §10 ТЗ).

### 2.3. Артефакты

- Финальное ТЗ с подписями всех стейкхолдеров.
- Закрытые ответы на 7 вопросов §13 ТЗ.
- Outlines пилота: регион, список ~150–200 ТТ, метрики baseline, KPI запуска.
- Готовая dev/staging-инфраструктура middleware.
- ROI-модель с целевыми коридорами.
- Список SKU MVP, MML по типам ТТ, план эталонной фотосъёмки.

**Критерии готовности к фазе 1.** ТЗ подписано, регион утверждён, dev-environment работоспособен, все 7 вопросов §13 имеют утверждённые ответы.

### 3.2. Команда v1

- 1 PM, 1 BA, 1 архитектор;
- 2 mobile dev (RN/Expo);
- 2 backend dev (DSD-middleware);
- 1 QA;
- 1 UX-дизайнер;
- 1 data engineer (BI).

### 3.3. Критерии готовности к пилоту

- Все KPI из `KPI_merchandising_audit.xlsx`, относящиеся к категории «опрос», покрыты вопросами; все вопросы связаны с KPI-кодами из таблицы §5.4 ТЗ.
- В backoffice заведены шаблоны для 4 типов ТТ: `retail`, `kiosk`, `horeca_bar`, `horeca_cafe`; есть ≥1 опубликованная версия на каждый тип.
- Sync push/pull работает offline → online без потерь и дубликатов; идемпотентность фото-загрузки подтверждена тестом повторной отправки.
- Загрузка 100 фото с одного устройства за визит проходит на 4G ≤ 5 мин; одиночное фото ≤ 5 МБ — ≤ 10 с (NFR §9).
- PSS считается одинаково локально (`dual`) и на сервере (расхождение ≤ 0,5%); `formula_version` фиксируется в `kpi_results`.
- Все NFR §9 пройдены на референсном устройстве среднего класса (камера ≤ 1 с, QG ≤ 0,5 с, KPI ≤ 90 с в 95% случаев).
- InfoSec-ревью архитектуры пройдено; антифрод-проверки (GPS/EXIF/sha256) активны.
- Доступность тестовой среды middleware ≥ 99,5% за 7 дней до пилота.

### 3.4. Пилот в одном регионе

- 5–10 мерчендайзеров, 1 супервайзер, ~150–200 ТТ в неделю.
- Длительность пилота: 6–8 недель.
- Параллельный «контрольный» бумажный чек-лист на части ТТ (~20%) для сверки.
- Еженедельные ретро с командой мерчендайзеров.
- KPI пилота:
  - время аудита одной ТТ ≤ 70% от бумажного процесса;
  - доля фото, прошедших Quality Gate с первого раза, ≥ 85%;
  - расхождение PSS app vs ручной чек-лист ≤ 5 п.п. на 80% визитов;
  - NPS мерчендайзеров ≥ +20.

### 3.5. Решение о масштабировании v1

После пилота — gate-review по критериям выше. Возможные исходы:
- **GO** — раскатка v1 на ещё 2–3 региона (rolling release).
- **FIX** — итерация багфиксов 4–6 нед, второй пилот.
- **NO-GO** — пересмотр архитектуры/UX (маловероятно при заранее согласованном ТЗ).

## 4. Фаза 2 — v2 «Trax SDK / открытый вендор» (4–6 мес)

**Цель.** Снять с мерчендайзера 50% ручного ввода: фейсинги, SoS, SoC, наличие POSM, цена с ценника — автозаполняются результатом распознавания.

### 4.1. Подготовка (параллельно с пилотом v1)

- Запуск **RFP** на CV-вендора. Кандидаты: Trax, Vispera, ParallelDots ShelfWatch, EYC, локальные игроки. Критерии: точность на тестовом наборе, стоимость на визит, поддержка кириллицы OCR, on-prem-возможности, SLA, опыт в FMCG.
- Выбор вендора → подписание контракта.
- Подготовка эталонных фото SKU и POSM (≥ 3 ракурса × ≥ 100 SKU MVP).

| --- | --- | --- | --- |

### 4.3. Команда v2 (дельта к v1)

- +1 ML/Computer Vision инженер (вендор-фокус);
- +1 backend dev (интеграция с CV API, очереди);
- +0,5 frontend (web) на раздел модерации.

### 4.4. KPI запуска v2

- Доля автоматически закрытых вопросов: ≥ 50% от вопросов с типом фото.
- mAP@0.5 на топ-SKU: ≥ 0,85.
- OCR accuracy цены: ≥ 0,92.
- Время аудита одной ТТ: ≤ 50% от бумажного процесса.
- Расхождение OSA (app vs ручной пересчёт): ≤ 5 п.п.
- Доля фото на ручную модерацию: ≤ 20% (постепенно к 10%).

### 4.5. Risk-register v2

| Риск | Митигатор |
| --- | --- |
| Вендор закрывает регион / повышает цену | Контракт с переходным периодом 6 мес; готовый адаптер для альтернативы |
| Низкое качество распознавания на наших SKU | Ручная разметка + дообучение со стороны вендора (контрактно) |
| Высокая латентность cloud | Edge-кэш на устройстве; для критичных KPI — локальный fallback |
| Зазеркаливание холодильника | Поляризационный фильтр + специальный режим съёмки |
| Конкуренты с похожей упаковкой | Контракт требует hard negatives в обучении |

## 5. Фаза 3 — v3 «Собственный CV-стек on-prem» (9–12 мес)

**Цель.** Снять зависимость от внешнего вендора, обучить модели под собственный SKU-каталог, получить полный контроль над качеством и стоимостью.

### 5.1. Этапы

| Этап | Длительность | Результат | Привязка к ТЗ |
| --- | --- | --- | --- |
| 3.1. Развёртывание ML-инфраструктуры on-prem | 8 нед | k8s namespace с RBAC и сетевыми политиками; GPU-ноды (NVIDIA T4 / A10 / L4); Triton Inference Server; MinIO/S3 для фото и эталонов; RabbitMQ или Kafka для `ml_jobs`; MLflow (реестр моделей и метрик); DVC + own catalog (feature store эталонов SKU/POSM); Airflow/Prefect (оркестрация переобучения) | §7.3, §10 |

## 6. Сквозные процессы

### 6.1. Управление качеством

- Еженедельные отчёты: точность KPI, доля Quality-Gate-rejects, доля фото на модерацию, расхождения human vs ML.
- Ежемесячные business reviews со стейкхолдерами.
- Ретро с мерчендайзерами раз в 2 недели в пилотах.

### 6.2. Релиз-стратегия мобильного приложения

- Все фазы — единый код, версии включаются флагами `merchandisingMlMode = survey | trax | cv`.
- Поэтапный rollout по регионам через настройки backoffice (без store-релиза приложения).
- Native config plugin `withMerchPlugins` обновляется только при смене SDK (v2/v3) — раз в квартал максимум.
- Поддержка минимум 2-х последних мажорных версий мобильного клиента (offline-устройства синхронизируются по графику).

### 6.3. Обучение мерчендайзеров

- v1: 1 онлайн-тренинг (1,5 ч) + видеогайд + чек-лист «как сделать хорошее фото».
- v2: добавляется блок «как доверять ML» (что подтверждать, что корректировать).
- v3: только notify «сменилась модель — точность улучшилась».

### 6.4. Поддержка

- L1 — служба поддержки DSD (общий процесс DSD Mini).
- L2 — команда модуля (mobile + backend dev rotation).
- L3 — ML-команда (только v2/v3).
- SLA пилотов: реакция ≤ 4 ч в рабочие часы; критичные баги — день в день.

### 6.5. Безопасность и соответствие

- Привлечение InfoSec на каждой фазе (ревью архитектуры, пентест).
- DPIA (если применимо) для фото с лицами/персоналом; опциональный blur лиц в backoffice.
- Аудит соответствия 152-ФЗ перед каждым крупным rollout; GDPR — по применимости.
- Антифрод-проверки: совпадение GPS визита с координатой ТТ ± `geofenceRadiusM`, EXIF GPS ≈ GPS устройства, уникальность `hash_sha256` фото в проекте (детекция повторной загрузки).
- Хранение фото ≥ 18 мес (настраивается); шифрование на диске (S3 SSE-KMS / MinIO encryption); TLS 1.2+; токены и ключи — `expo-secure-store`.
- RBAC: `merchandiser`, `supervisor`, `marketer`, `moderator`, `admin`; SSO с существующей системой; аудит-лог действий пользователей и модераторов.

### 6.6. Соответствие NFR (§9 ТЗ)

Каждая фаза проходит NFR-валидацию перед gate-review. Контрольные значения:

| Показатель | Цель | Когда проверяем |

| Quality Gate | ≤ 0,5 сек | v1 |

## 7. Ключевые ограничения и допущения

- **On-prem ML — обязательное условие v3.** Облако допустимо только для обучения и stress-test, inference — on-prem.
- **Вендор v2 открыт.** В коде заложен интерфейс `MerchVisionAdapter`; финальный выбор — после RFP.
- **Пилот — один регион РФ.** Расширение по регионам — только после gate-review v1.
- **Российский рынок.** OCR на кириллице, валюта — рубль, соответствие 152-ФЗ, локализация ru как primary, en — secondary.
- **Минимальные устройства.** Android 10+ / iOS 14+, камера ≥ 8 Мп, 3 ГБ ОЗУ. Девайсы ниже минимума не допускаются к аудиту.
- **Существующие SFA, master-data, planogram системы** предоставляют стабильные REST-API. Если каких-то систем нет — они закрываются ручным наполнением в backoffice (раздел Master Data).
- **Эталоны SKU.** ≥ 3–5 ракурсов на SKU. Стартовый каталог MVP — 80 SKU, год 1 — 500 SKU.
- **Миграция схемы БД v6 — forward-only.** Расширения `visit_reports` (7 колонок) и новые таблицы `audit_templates`/`audit_answers`/`audit_photos`/`ml_jobs`/`kpi_results`, плюс `products.is_mml`/`mml_priority` добавляются через идемпотентный `ensureColumns`. **Откат к старой схеме не предусмотрен.** Если приложение откатить на старую версию, новые колонки/таблицы останутся в БД, но игнорируются (forward-compatible). Полное удаление новых объектов — только через ручной DROP в backoffice, либо через `clearAllData()` + переинициализацию.

## 9. Контрольные точки и решения

- **GR-0** (после фазы 0): ТЗ согласовано, регион выбран → старт v1.
- **GR-1** (после пилота v1): KPI пилота достигнуты → мульти-региональный rollout v1 + старт RFP v2.
- **GR-2** (после RFP v2): вендор выбран → старт фазы 2.
- **GR-3** (после пилота v2): автозаполнение ≥ 50%, mAP ≥ 0,85 → rollout v2 + старт фазы 3.
- **GR-4** (после shadow-сравнения v3 vs v2): паритет/превосходство v3 → переключение регионов на v3.
- **GR-5** (год 1 v3): все целевые KPI достигнуты → закрытие проекта внедрения, переход в стадию поддержки и регулярного дообучения.
